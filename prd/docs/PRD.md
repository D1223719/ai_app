# AI PRD: AI 履歷顧問機器人 (AI Resume Consultant Agent)

### 1. 專案概述 (Overview)
- **產品名稱:** AI 履歷顧問機器人
- **背景與痛點:** 求職者針對不同職缺 (JD) 客製化履歷耗時且難以掌握重點。傳統工具僅提供單向的文字潤飾，缺乏根據特定職缺進行脈絡分析、且無法反覆修改與討論的互動性。
- **業務目標 (Business Metrics):**
  - 提升求職者的履歷投遞準備效率。
  - 提升用戶留存率：促使用戶在每次應徵新職缺時，都會開啟新的對話 Session。

### 2. 用戶場景 (User & Use Cases)
- **目標用戶:** 正在積極求職或準備轉職的專業人士。
- **核心場景 (User Story):** 作為一個求職者，我希望可以上傳我現有的履歷文件與目標職缺的截圖，讓 AI 顧問透過多輪對話協助我修改經歷描述，並自動將適合的關鍵字融入履歷中，以便於我能快速產出高匹配度且專業的履歷。

### 3. AI 功能與 UX 設計 (AI Features & UX)
基於對話式代理 (Conversational Agent) 架構，本系統具備以下核心功能：

- **上傳圖片或文件 (Multimodal Input):** 支援上傳 PDF/Word (基礎履歷) 與 圖片/PDF (目標職缺 JD)，作為對話上下文的參考基礎，免除手動複製貼上。
- **對話狀態管理與訊息系統:** 採用對話式 UI，設計完整 message 結構 (role, content, timestamp)。支援多輪對話，讓用戶可以不斷下達修改指令 (如：「幫我縮短這段」、「語氣更自信一點」)。
- **對話歷史管理 (Sessions):** 側邊欄設計，每個職缺對應一個獨立的 Chat Session (例如："Google 產品經理"、"Line 前端工程師")，方便用戶隨時切換與追蹤。
- **記憶機制 (Memory):** 建立 User Profile，跨 Session 記住用戶的年資、核心技能或偏好語氣，減少重複自我介紹。
- **回答控制 (Response Control):** 提供「重新生成 (Regenerate)」按鈕，讓用戶可以快速獲取不同版本的修改建議；並提供「中止回應」按鈕以節省等待時間。
- **工具整合 (Tool Use):** 
  - **Web Search API:** AI 可主動呼叫搜尋工具查找該應徵企業的「核心價值/文化」，並以此作為履歷修改的風格參考。
  - **PDF Export API (規劃中):** 允許對話結束後，將最終確認的履歷內容排版並匯出成 PDF。

### 4. 數據與模型需求 (Data & Model Requirements)
- **基礎模型:** 需具備長文本理解、多模態圖片解析與強大指令遵循 (Instruction Following) 能力的大型語言模型 (如 Gemini 1.5 Pro)。
- **評估指標 (Evaluation Metrics):**
  - **對話連貫性:** 在大於 5 輪以上的對話中，AI 是否仍能記住最初始上傳的履歷內容。
  - **回應延遲 (Latency):** 單次修改建議的回覆時間應 < 3 秒，確保對話體驗流暢。

### 5. 邊界情況與限制 (Edge Cases & Limitations)
- **技術架構限制:**
  - 前端限定使用原生 HTML / Vanilla JS / CSS 進行開發，不使用大型框架。
  - 後端限定使用 Python FastAPI 搭配 SQLite 輕量資料庫。
- **已知限制:** 若上傳的履歷排版極度複雜 (如雙欄位且充滿密集圖表)，影像解析或文本擷取可能會遺漏資訊。
- **極端狀況處理:**
  - **缺乏輸入:** 用戶輸入過於簡短 (如：「幫我寫履歷」但未提供任何背景) 時，AI 應啟動反問機制，引導用戶上傳資料或回答基礎問題。
  - **幻覺防護 (Guardrails):** 系統需在 System Prompt 中嚴格限制「絕對不允許無中生有未經歷過的技能或專案數據」。

### 6. 安全、隱私與合規 (Security & Privacy)
- **隱私考量:** 履歷包含大量個人識別資訊 (PII，如電話、地址)。上傳文件前應在前端提示用戶遮蔽敏感個資，系統端也應確保對話歷史資料與文件受到加密保護，且不被用於未授權的第三方模型訓練。
